## 6. 大模型参数传递混乱——系统性根因分析

### 6.1 量化现状：LLM 看到了什么

| 指标 | 当前值 | 行业参考（简单 MCP 服务） |
|---|---:|---:|
| MCP 暴露工具数 | **28** | 5~10 |
| 写入口工具数 | **5**（apply_visual_actions / submit_unity_task / apply_script_actions / set_ui_properties / set_serialized_property） | 1~2 |
| Schema 发现工具数 | **4**（get_action_schema / get_tool_schema / get_write_contract_bundle / get_action_catalog） | 0~1 |
| L3 注册 action type 数 | **~40**（含 13 个 deprecated alias） | N/A |
| 写操作最小步骤数 | **3**（get_current_selection → apply_visual_actions → get_unity_task_status 轮询） | 1 |
| `action_data` 内部字段在工具 schema 中的可见性 | **完全不可见**（`type: "object"`，无 properties） | 内联可见 |
| 锚点组合规则数 | **4 种**（target_required / parent_required / target_and_parent_required / target_or_parent_required） | N/A |
| 单次 `apply_visual_actions` 的 JSON 嵌套深度 | **4 层**（envelope → actions[] → action → action_data） | 1~2 层 |

### 6.2 根因判定：不是"大模型不理解"，是"设计上认知负荷超限"

**结论先行：当前问题的根本原因是 MCP 协议面的信息架构设计缺陷，不是大模型能力不足。**

以下为系统性归因，按影响权重排序：

#### 根因 1：`action_data` 是黑盒——LLM 在 schema 中看不到要填什么（影响权重：35%）

```
// apply_visual_actions 的 inputSchema 中：
action_data: {
  type: "object",                    // ← 只告诉 LLM "这是一个对象"
  description: "Open payload object" // ← 没有 properties，没有 required
}
```

**这是最核心的设计缺陷。** 当 LLM 调用 `apply_visual_actions` 想做一个 `rename_object` 时：
- Schema 告诉 LLM `action_data` 是 `type: "object"`——但里面该放什么字段？放 `name`？`new_name`？`object_name`？**完全没线索。**
- LLM 必须**先**调 `get_action_schema` 查出 `rename_object` 的 `action_data` 需要 `{name: "..."}` 字段——这又多了一轮 tool call。
- 即使 LLM 查了 schema，也必须在上下文中**跨两个 tool call 结果拼装** JSON——这对当前所有主流 LLM 都是高出错概率操作。

**对比标准 MCP 设计**：好的 MCP 工具会把所有必填字段直接内联在 `inputSchema.properties` 中，LLM 一次就能看到完整的契约，不需要"先查再用"。

#### 根因 2：三层锚点体系 + 条件必填规则——LLM 难以记住哪个 action 需要哪种锚点（影响权重：25%）

当前锚点体系：

| 锚点 | 层级 | 必填性 |
|---|---|---|
| `write_anchor` | 顶层 envelope | **始终必填**（`object_id` + `path`） |
| `target_anchor` | 每个 action | **条件必填**——取决于 action type |
| `parent_anchor` | 每个 action | **条件必填**——仅 `create_object` / `set_parent` 需要 |

**问题本质**：锚点的必填规则是 **per-action-type 的条件逻辑**，但 JSON Schema 的 `anyOf: [{required: ["target_anchor"]}, {required: ["parent_anchor"]}]` **无法表达这个条件关系**。LLM 只能从 description 文本中的"Anchor decision table"去推断，而这段文本极易被忽略或误解。

常见失败模式：
- `rename_object` 时传了 `parent_anchor` 而不是 `target_anchor`
- `create_object` 时只传了 `target_anchor` 没传 `parent_anchor`
- `set_parent` 时只传了一种锚点而不是两种都传
- 锚点中 `object_id` 或 `path` 漏填一个

#### 根因 3：工具面膨胀——28 个工具远超 LLM 有效管理阈值（影响权重：20%）

| 分类 | 工具 | 数量 |
|---|---|---|
| 写操作 | apply_visual_actions / apply_script_actions / submit_unity_task / set_ui_properties / set_serialized_property | 5 |
| 读/感知 | get_current_selection / get_hierarchy_subtree / get_gameobject_components / get_scene_roots / find_objects_by_component / query_prefab_info / list_assets_in_folder / get_ui_tree / get_ui_overlay_report / get_serialized_property_tree / capture_scene_screenshot | 11 |
| Schema 发现 | get_action_catalog / get_action_schema / get_tool_schema / get_write_contract_bundle | 4 |
| 状态管理 | get_unity_task_status / cancel_unity_task | 2 |
| 诊断 | validate_ui_layout / hit_test_ui_at_viewport_point / hit_test_ui_at_screen_point / preflight_validate_write_payload | 4 |
| 设置 | setup_cursor_mcp / verify_mcp_setup | 2 |
| **合计** | | **28** |

**MCP 最佳实践建议工具数在 5~10 个。** 28 个工具意味着：
- `tools/list` 返回的 JSON 非常庞大（估算 15K~20K tokens）
- LLM 的注意力被严重稀释，尤其对相似命名的工具（如 5 个写工具）容易选错
- 4 个 schema 发现工具本身就形成了"先要知道查哪个工具，再查参数"的二阶复杂度

#### 根因 4：写操作是异步多步流程——每个环节都可能出错（影响权重：10%）

最小写操作需要 **3 步 tool call**：
1. `get_current_selection` → 获取 `read_token` + `object_id` + `path`
2. `apply_visual_actions` → 构造 envelope + action + action_data → 获取 `job_id`
3. `get_unity_task_status(job_id)` → 轮询直到 `succeeded/failed/cancelled`

**问题**：
- 步骤 1 返回的 `read_token` 是临时凭证，一旦 stale 需要重新获取
- 步骤 2 返回 `accepted/queued` 只是提交态，**不是成功态**——LLM 经常误以为 `accepted` 就是完成
- 步骤 3 需要轮询——LLM 可能只查一次就结束

**对比**：简单的 MCP 工具设计通常是**同步返回结果**，单个 tool call 即完成。

#### 根因 5：Deprecated Alias 污染——13 个废弃名仍然可用（影响权重：5%）

| Canonical | Deprecated Alias |
|---|---|
| `create_object` | `create_gameobject` |
| `set_active` | `set_gameobject_active` |
| `rename_object` | `rename_gameobject` |
| `destroy_object` | `destroy_gameobject` |
| `set_local_position` | `set_transform_local_position` |
| `set_local_rotation` | `set_transform_local_rotation` |
| `set_local_scale` | `set_transform_local_scale` |
| `set_world_position` | `set_transform_world_position` |
| `set_world_rotation` | `set_transform_world_rotation` |
| `set_rect_anchored_position` | `set_rect_transform_anchored_position` |
| `set_rect_size_delta` | `set_rect_transform_size_delta` |
| `set_rect_pivot` | `set_rect_transform_pivot` |
| `set_rect_anchors` | `set_rect_transform_anchors` |
| ... | + 其他 alias |

两套名字都能工作但行为微妙不同（特别是 `create_object` vs `create_gameobject` 在不同 L2/L3 校验路径上），增加了 LLM 的决策不确定性。

#### 根因 6：错误反馈虽结构化但修复路径仍需推理（影响权重：5%）

V2-PROTOCOL 修复后，错误响应已包含 `field_path`、`fix_kind`、`suggested_patch`。但：
- `corrected_payload` 不是所有错误都能生成（L3 来源错误仅 best-effort）
- LLM 需要理解 JSON Patch 语义（`op: "replace", path: "/actions/0/target_anchor"`）
- 对 `E_STALE_SNAPSHOT` 类错误，LLM 需要知道"重新获取 read_token 再重试"的流程

### 6.3 因果链图

```
LLM 收到用户指令 "把 Panel 改名为 Header"
    │
    ├─① 不知道该调哪个写工具（5 个写工具）
    │     ↓ 可能选错 submit_unity_task 而非 apply_visual_actions
    │
    ├─② 不知道 action_data 里放什么（schema 是黑盒）
    │     ↓ 猜测用 {new_name: "Header"} 而非 {name: "Header"}
    │
    ├─③ 不知道该用 target_anchor 还是 parent_anchor
    │     ↓ 可能传空对象 {} 或传错锚点类型
    │
    ├─④ 忘记先获取 read_token
    │     ↓ 直接调写工具 → E_SCHEMA_INVALID
    │
    ├─⑤ 锚点中漏了 object_id 或 path
    │     ↓ E_ACTION_SCHEMA_INVALID
    │
    └─⑥ 以为 accepted 就是成功，不轮询终态
          ↓ 用户看到"操作完成"但 Unity 没变
```

**每个环节都可能失败，且失败后的修复路径又引入新的 tool call 和新的出错机会。** 这就是为什么一个简单的 rename 操作会出现 3~5 次重试。

### 6.4 V2-PROTOCOL 修复方案的效果评估

| 修复措施 | 解决了什么 | 没解决什么 |
|---|---|---|
| 错误引导纠偏（Phase A） | 锚点错误不再误导到 action-only schema | 不减少出错概率，只减少修复往返 |
| 最小模板（Phase B） | LLM 可查到可执行模板 | 仍需额外 tool call 去查，schema 本身不含 |
| 预检 preflight（Phase C） | 可在不执行的情况下校验 | 增加了流程步骤（变成 4 步而非 3 步） |
| 重试熔断（Phase D） | 阻断无效重复重试 | 不减少首次出错概率 |
| 黄金模板 + 决策表（Phase G） | LLM 可查到锚点决策规则 | 规则在 description 文本中，不在 schema 结构中 |
| SSOT 契约统一（Phase G） | L2/L3 门禁不再漂移 | 不解决 schema 信息不可见的根问题 |

**总体评估**：V2-PROTOCOL 修复把"修复成本"降低了（错误后恢复更快），但没有降低"首次出错概率"。根本原因是 **MCP 工具面的信息架构没有变**——`action_data` 仍然是黑盒，28 个工具仍然全部暴露，三步异步流程仍然不变。

### 6.5 根本性解决方向（不含实施方案）

| 方向 | 原理 | 预期效果 |
|---|---|---|
| **A. 扁平化写工具 schema** | 将高频 action 的 `action_data` 字段直接内联到 tool schema 的 properties 中，消灭黑盒 | LLM 在 tools/list 时就能看到所有必填字段 |
| **B. 大幅缩减工具数** | 合并同类工具（如 5 个写工具合并为 1~2 个；4 个 schema 发现工具合并为 1 个）；对 LLM 不常用的工具默认不暴露 | 将工具数从 28 降到 8~12，降低注意力稀释 |
| **C. 同步返回模式** | 对简单写操作（单 action），L2 内部完成 submit + poll + 返回终态，对 LLM 暴露同步接口 | 写操作从 3 步降为 1 步 |
| **D. 消灭 deprecated alias** | 在 L2 入口层将所有 alias 统一归一化为 canonical，不再向 LLM 暴露双套命名 | 减少命名歧义 |
| **E. 锚点自推断 + 默认值** | 当 `write_anchor` 已提供且为单 action 时，自动将 `target_anchor` 设为 `write_anchor`（如果 anchor_policy 允许） | 减少锚点漏填错误 |

**优先级建议**：`A > B > C > D > E`

方向 A 是最高优先级——只要 `action_data` 不再是黑盒，LLM 的首次构造成功率就能从当前估计的 30~50% 提升到 70~90%（推断区间，需上线后用真实埋点验证）。

## 7. 结论摘要

- 当前工程主架构已具备稳定的三层职责与协议治理能力，主干质量可用。L2 全量 335 测试、R20 协议 86 测试全绿。
- 冗余主要集中在：历史文档、兼容残留壳层、纯测试与验收资产。
- **大模型参数传递混乱的核心原因是 MCP 工具面设计的认知负荷超限**，不是大模型能力不足：
  - `action_data` 在 schema 中是黑盒（无 properties 定义）
  - 28 个暴露工具远超 MCP 最佳实践上限
  - 三层锚点 + 条件必填规则无法被 JSON Schema 结构化表达
  - 写操作强制 3 步异步流程
- V2-PROTOCOL 修复降低了"修复成本"但未降低"首次出错概率"。
- 若目标是根本解决 LLM 参数混乱问题，最优先方向是 **扁平化写工具 schema（消灭 action_data 黑盒）** 和 **大幅缩减暴露工具数**。
