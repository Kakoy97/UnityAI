## 第二重排毒：L2 与 L3 的"阴阳校验"对齐 (Validation Unification)

### 2.1 核心矛盾：`resolveVisualActionField` — L2 的"模糊通行证"

**代码位置**：`_mcpWriteValidatorsImpl.js:93-103`

```javascript
function resolveVisualActionField(action, fieldName) {
  if (action[fieldName] !== undefined) {     // ← 先查 action 顶层
    return action[fieldName];
  }
  if (isObject(action.action_data) && action.action_data[fieldName] !== undefined) {
    return action.action_data[fieldName];    // ← 再查 action_data 内部
  }
  return undefined;
}
```

**这个函数是整个阴阳问题的根源**。它让 L2 对以下两种写法都放行：

| 写法 A（规范格式） | 写法 B（模糊格式，L2 默许通过） |
| --- | --- |
| `{type:"create_object", action_data:{name:"Foo"}}` | `{type:"create_object", name:"Foo"}` |

### 2.2 隐式转换链条（写法 B 能走通的完整暗道）

```
LLM 发送: {type:"create_object", name:"Foo", parent_anchor:{...}}
    │
    ▼ L2 validateVisualActionHardcut()
    │  resolveVisualActionField(action, "name") → 命中 action.name → 通过 ✓
    │
    ▼ L2 turnPayloadBuilders.resolveVisualActionData()
    │  action.action_data 不存在 → 走 buildLegacyVisualActionData()
    │  buildLegacyVisualActionData() 遍历 action 的所有 key：
    │    skipKeys = {type, target_anchor, parent_anchor, action_data, ...}
    │    "name" 不在 skipKeys 中 → 被收集进 action_data → {name: "Foo"}
    │
    ▼ L2 序列化为 action_data_json + action_data_marshaled
    │
    ▼ L3 VisualActionContractValidator.TryResolveFieldValue()
    │  先查 VisualLayerActionItem.name（直接字段，但此时为空因为 name 不在 DTO 顶层传输）
    │  再查 action_data_json 解析出的 dict → 命中 {name: "Foo"} → 通过 ✓
```

**结论**：写法 B 之所以能工作，**完全依赖** `buildLegacyVisualActionData` 这个隐式转换器。一旦去掉这个兼容层，所有使用写法 B 的调用在 L3 端静默失败。

### 2.3 具体字段 L2/L3 差异矩阵

| 字段 | L2 验证 | L3 验证 | 差异判定 |
| --- | --- | --- | --- |
| `name` (create_object) | `resolveVisualActionField` 双源查找 | `TryResolveFieldValue` 查直接字段 + action_data_json | 🟡 表面一致但依赖隐式转换 |
| `name` (rename_object) | 同上 | `RenameGameObjectActionData.name` 仅从 action_data_json 反序列化 | 🔴 阴阳 |
| `component_assembly_qualified_name` | 同上双源查找 | `AddComponentActionData` 仅从 action_data_json 反序列化 | 🔴 阴阳 |
| `active` (set_active) | 无显式验证（bool 字段不检查） | `SetGameObjectActiveActionData.active` 仅从 action_data_json 反序列化 | 🔴 阴阳：顶层 active=true 不经转换则 L3 得到默认 false |
| `target_anchor` | 要求 `object_id` + `path` 都非空 | 要求 `object_id` + `path` 都非空 | ✅ 一致 |
| `parent_anchor` | 同上 | 同上 | ✅ 一致 |
| `anchor_policy` 默认值 | fallback: create → parent_required, 其他 → target_or_parent_required | 同等 fallback 逻辑 | ✅ 基本一致 |

### 2.4 开刀方案：L2 绝对向 L3 严格对齐

**原则：彻底删除 `resolveVisualActionField` 的双源查找逻辑，L2 只检查 `action.action_data.{field}` 路径。**

| 步骤 | 操作 | 影响面 |
| --- | --- | --- |
| S1 | 将 `resolveVisualActionField` 改为**仅**查 `action.action_data[fieldName]` | `_mcpWriteValidatorsImpl.js` 约 13 处调用 |
| S2 | 在 `validateVisualActionHardcut` 中，对已知 actionType 检查必填字段是否在 `action_data` 中存在 | 同上 |
| S3 | 删除 `turnPayloadBuilders.js` 的 `buildLegacyVisualActionData()` 函数 (~30 行) | `turnPayloadBuilders.js` |
| S4 | 修改 `resolveVisualActionData()` 使其仅接受 `action.action_data` 对象 | `turnPayloadBuilders.js:182-196` |

## 第三重排毒：扁平化参数结构的改造准备 (Schema Flattening Prep)

### 3.1 当前最容易出错的写工具分析

**`apply_visual_actions`** 是 LLM 参数混乱的重灾区。当前 schema：

```
apply_visual_actions                             ← 深度 0
├── based_on_read_token: string                  ← 深度 1 ✓
├── write_anchor: {object_id, path}              ← 深度 1→2
├── actions: array                               ← 深度 1
│   ├── [0].type: string                         ← 深度 2
│   ├── [0].target_anchor: {object_id, path}     ← 深度 2→3
│   ├── [0].parent_anchor: {object_id, path}     ← 深度 2→3
│   └── [0].action_data: { ??? }                 ← 深度 2→3 ■■■ 黑盒 ■■■
│       └── name / x / y / z / ...               ← 深度 3→4
```

**最大嵌套深度：4 层**。`action_data` 是 `{type: "object"}` 无 properties 定义的黑盒。

### 3.2 对照组：`set_ui_properties` 的正确示范

`set_ui_properties` 成功率高得多，因为**每个字段都有明确 schema 定义**（rect_transform、image、text、layout_element 各自有完整 properties 和 required）。LLM 不需要猜。

### 3.3 改造方案评估（本轮仅评估，不落地）

#### 方案 A：拆分为专用扁平工具（最优但违反"不做加法"）

为高频 action types 各创建一个扁平化专用 MCP 工具，最大嵌套深度降至 1 层。**本轮不实施**。

#### 方案 B：在现有工具中内联 action_data schema（推荐本轮实施）

修改 `apply_visual_actions` 的 `actions[].action_data` 定义，从：
```json
{ "type": "object" }
```
改为按 actionType 使用 `oneOf` 显式定义每种 action_data 的 properties 和 required。

**这属于"减法"** — 减掉 schema 的模糊度，不引入新逻辑。可在第二重 S5 步骤中合并实施。

### 3.4 L2 组装层改动评估（方案 A 参考，本轮不做）

如选方案 A，每个专用工具约需 130 行（definition+handler+validator），L2 handler 组装完整 `actions[]` 后复用现有 turnService 通道，**L3 零改动**。
