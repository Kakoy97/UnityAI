## 1. L1 协议面（API & Schema Layer）

### 1.1 当前边界定义

L1 的实际物理边界：
- 工具定义与清单：`sidecar/src/mcp/commands/definitions/index.js`、`sidecar/src/mcp/commands/legacyCommandManifest.js`、`sidecar/src/mcp/commandRegistry.js`。
- MCP 暴露与可见性：`sidecar/src/mcp/mcpServer.js`。
- 外部写入参门禁：`sidecar/src/domain/validators/_mcpWriteValidatorsImpl.js`。

当前对外协议形态：
- `tools/list` 固定使用 compact schema（`mcpServer.js` L280-L288，`toolsListSchemaMode: "compact"`）。
- full schema 需单独调用 `get_tool_schema`（`commandRegistry.js` L411-L413，`mcpServer.js` L605）。
- 写工具入口确实并存，但需精确到行：
  - `submit_unity_task`（`definitions/index.js` L4）
  - `apply_script_actions`（L10）
  - `apply_visual_actions`（L11）
  - `set_ui_properties`（L12）
  - `set_serialized_property`（L13）
  - `preflight_validate_write_payload`（L18）

### 1.2 偏离蓝图的现状

- `set_serialized_property` 生命周期仍是 `experimental`（`definitions/set_serialized_property.js` L50），与路线图“泛化写已收口”的叙事存在落差（ROADMAP L42-L47）。
- `preflight_validate_write_payload` 的 `tool_name` 枚举仅 3 项，不含 `set_serialized_property`（`definitions/preflight_validate_write_payload.js` L68-L72；`mcpEyesWriteService.js` L689-L693）。
- 写工具在 `tools/list` 下是裁剪视图，不是完整视图：
  - 写工具裁剪参数：`maxDepth: 3`、`maxPropertiesPerObject: 8`、`maxCombinatorItems: 3`（`commandRegistry.js` L231-L236）。
  - 这意味着“查到了 schema”不等于“拿到了完整 schema”。

### 1.3 深层次矛盾定位

核心矛盾是“对外单一协议外观”与“内部多契约分层”并存：
- LLM 首先看到的是裁剪后的 schema；完整契约、action_data 细节、写入约束需要二次甚至三次查询拼接。
- 协议入口本身已经携带语义差异（原始任务提交通道 vs 语义封装通道），接口选择成为行为变量。
- 结果不是“业务能力不足”，而是“协议信息充分性不稳定”。

## 2. L2 网关层（Sidecar / Node.js Layer）

### 2.1 当前边界定义

L2 的实际物理边界：
- 路由：`sidecar/src/api/router.js`。
- 协调：`sidecar/src/application/turnService.js`。
- 任务网关：`sidecar/src/application/mcpGateway/mcpGateway.js`。
- 写编排：`sidecar/src/application/mcpGateway/mcpEyesWriteService.js`。
- 错误反馈：`mcpErrorFeedback.js`、`unityReportNormalizer.js`。

当前结构事实：
- Router 先走 registry 分发，再走手写路由分支（`router.js` L89-L99 与 L102-L190）。
- `TurnService` 同时承接直提交通道和封装通道（`turnService.js` L507-L539）。
- `submitUnityTask` 本身集成 OCC、幂等、队列冲突、入库、状态推进（`mcpGateway.js` L231-L409）。

### 2.2 偏离蓝图的现状

- `dry_run` 不存在单一行为：
  - `apply_script_actions` 与 `set_ui_properties` 在 L2 直接短路（`mcpEyesWriteService.js` L227-L239、L438-L451）。
  - `apply_visual_actions` 存在条件分叉：仅当 `dry_run=true` 且 `!shouldDispatchDryRunToUnity(actions)` 才短路（L332-L353）；否则继续提交到 Unity（L355-L363）。
  - `shouldDispatchDryRunToUnity` 条件是“全部 action 均为 `set_serialized_property`”（L1022-L1035）。
- `dry_run` 虽标记 deprecated alias，但仍是活跃治理路径（L113-L118、L1432-L1439、L1526-L1536）。

### 2.3 深层次矛盾定位

L2 的核心矛盾是“统一门禁目标”与“路径分叉执行现实”：
- 同一写意图可以走多条入口，并在不同入口触发不同前置序列。
- `mcpEyesWriteService` 内部存在明显重复链路：
  - `applyScriptActions`：L161-L252（约 92 行）
  - `applyVisualActions`：L254-L365（约 112 行）
  - `setUiProperties`：L367-L478（约 112 行）
  - `runWritePreflightForTool`：L655-L779（约 125 行）
- 四段都重复“握手 -> Unity ready -> read_token -> schema -> precondition -> 结果封装”的同构序列，形成代码级重复，而非仅概念级分叉。

代表性失败路径（仅描述现状）：
- `apply_visual_actions(dry_run=true)` 若 action 集合不全是 `set_serialized_property`，在 L2 即短路返回（L332-L353）。
- 同样的 `dry_run=true` 若 action 集合全是 `set_serialized_property`，则继续提交到 L3（L355-L363 + L1022-L1035）。
- 同一参数位（`dry_run`）在同一路由上具备双语义执行路径。

### 4.1 当前边界定义

关键接缝位于：
- Action 数据线缆：`turnPayloadBuilders.js`、`_mcpWriteValidatorsImpl.js`、`SidecarContracts.Action.cs`、`VisualActionContractValidator.cs`、`McpVisualActionContext.cs`。
- Query 数据线缆：`SidecarContracts.Query.cs`、`IUnityQueryHandler.cs`。

### 4.2 偏离蓝图的现状

- 对外字段已隐藏，但内部仍是多形态并存：对象 + JSON 字符串 + base64url 字符串。
  - L2 构建桥接（`turnPayloadBuilders.js` L199-L223）
  - L3 DTO 保留字符串字段（`SidecarContracts.Action.cs` L45-L46、L225-L226）
  - L3 校验/执行上下文重复解码（`VisualActionContractValidator.cs` L314-L320；`McpVisualActionContext.cs` L115-L126）
- Query 管线仍是双轨：`query_payload_json` 与 legacy `payload` 并存（`SidecarContracts.Query.cs` L87-L88、L93-L159；`IUnityQueryHandler.cs` L125-L137）。
- `set_serialized_property` 不是“直接写管线”，而是二次映射：
  - 先映射成 `apply_visual_actions` 的单 action（`set_serialized_property/validator.js` L411-L433）
  - 再通过 `turnService.applyVisualActionsForMcp` 提交（`set_serialized_property/handler.js` L42-L45）
  - 形成额外的编码/转发层。

### 4.3 深层次矛盾定位

纵向链路的病理机制：
- 单一业务语义在跨层传输中被多次编码与重解码。
- 同一错误在多层被重复归一，错误源与最终呈现不是单跳映射。
- Action/Query 双管线都保留历史兼容分支，系统处于长期叠态而非单态契约。

## 5. 结论性问题定界（严重度分级，仅问题）

### P0（根因级）
1. 协议外观单一，但契约信息被拆分到 compact schema / full schema / action schema 多入口，且默认入口是裁剪视图。  
2. 同一写意图存在多入口执行语义，入口选择可改变前置门禁与执行路径。  
3. 纵向链路中 Action 数据多形态并存（对象/JSON/base64url），跨层语义一致性依赖回退机制。

### P1（放大器级）
1. L2 写链路前置序列在多函数重复实现（约 400+ 行同构流程）。  
2. `dry_run` 在同一路由上存在条件分叉（短路 vs 转发），放大调用结果不确定性。  
3. 错误归一多点并行，降低“错误来源 -> 错误呈现”的可追踪性。

### P2（症状级）
1. 失败位置跨层漂移（校验失败 / 预执行失败 / 执行失败），诊断路径依赖链路细节。  
2. Query 双轨 payload 回退保留，导致协议演化完成态与运行态不一致。
