# 全栈大换血重构蓝图

## 0. 目标与硬约束

### 0.1 目标

1. 让 LLM 一次请求即可完成跨领域写操作，减少 token 失效和链路脆弱性。  
2. 让写请求默认返回终态（success/failed），避免长期 `pending` 黑洞。  
3. 保持契约白盒化，彻底禁用 `action_data*` 黑盒。  

### 0.2 不可破坏安全防线

1. 保留 `UndoGuard` 原子回滚。  
2. 保留 OCC（`based_on_read_token`）防并发冲突。  
3. 保留 Job 状态机与幂等键语义。

## 1. L1 协议层重构（强类型聚合事务）

### 1.1 工具形态

写侧统一为：

1. `execute_unity_transaction`（唯一写工具）  

读侧新增：

1. `get_scene_snapshot_for_write`（获取全局层次摘要 + 可用 `read_token`，不依赖 selection）  
2. `get_write_contract_v2`（返回 full schema + 示例）  

设计原则：

1. 使用根级具名数组（`hierarchy_mutations`/`property_mutations`/`layout_mutations`/`file_mutations`）。  
2. 不使用跨域巨型 `oneOf`。  
3. 单次 payload 跨领域组合，复用同一个 `read_token` 与同一个 Undo 事务。

### 1.4 `get_scene_snapshot_for_write` 契约

输入：空对象或可选 `scope_path`。  
输出至少包含：

1. `scene_revision`  
2. `read_token`（可用于后续写请求）  
3. `roots`（层次摘要，含 `object_id` + `path`）  
4. `unity_connection_state`

说明：该工具目标是“高稳定获取写前上下文”，但在 Unity 断连或硬故障时仍会失败，错误码需可诊断。

## 2. L2 网关层重构（线性服务流 + Sync-over-Async）

### 2.1 指令注册中心

保留命令注册中心，但写入口收敛为：

1. `execute_unity_transaction`  
2. `get_scene_snapshot_for_write`  
3. `get_write_contract_v2`

### 2.2 线性调用流（Controller -> Service）

```ts
export async function handleExecuteUnityTransaction(req: HttpRequest): Promise<HttpResponse> {
  const body = await parseJson(req);

  // 1) Schema + 枚举归一化（大小写宽容）
  const normalized = normalizeEnumsCaseInsensitive(body);
  const schemaCheck = validateSchema("execute_unity_transaction", normalized);
  if (!schemaCheck.ok) return badRequest(schemaCheck.error);

  // 2) 前置门禁
  const readiness = checkUnityReady();
  if (!readiness.ok) return readiness.outcome;

  const occ = checkReadToken(normalized.based_on_read_token);
  if (!occ.ok) return occ.outcome;

  const replay = tryIdempotentReplay(normalized.idempotency_key);
  if (replay.hit) return replay.outcome;

  const precondition = evaluatePreconditions(normalized.preconditions);
  if (!precondition.ok) return precondition.outcome;

  // 3) validate 模式短路
  if (normalized.execution_mode === "validate") {
    return ok(buildValidateReceipt(normalized));
  }

  // 4) 入队 + 调度
  const job = createOrQueueJob(normalized);
  await dispatchToUnity(job, toUnityPayload(normalized));

  // 5) Sync-over-Async: 默认阻塞等待终态（15s）
  const waited = await waitForTerminal(job.job_id, {
    timeout_ms: 15000,
    poll_interval_ms: 250
  });
  if (waited.terminal) {
    return waited.outcome; // success / failed / cancelled
  }

  // 6) 超时未终态：返回可诊断 pending（非黑盒）
  return acceptedPending({
    job_id: job.job_id,
    pending_reason: waited.pending_reason,      // e.g. unity_domain_reload
    last_stage: waited.last_stage,              // e.g. action_pending
    elapsed_ms: waited.elapsed_ms,
    next_poll_after_ms: 1000,
    watchdog_deadline_ms: 120000,
    can_cancel: true
  });
}
```

### 2.3 Pending 黑洞治理（必须项）

1. **同步等待窗口**：默认等待 15 秒，快操作直接返回终态。  
2. **watchdog**：超过 `watchdog_deadline_ms`（建议 120 秒）自动终止。  
3. **自动终止语义**：返回 `failed/cancelled`，错误码如 `E_UNITY_ACTION_STALLED`。  
4. **可诊断 pending 字段**：必须含 `pending_reason`、`last_stage`、`next_poll_after_ms`。  

### 2.4 大小写宽容（Resilience）

L2 Validator 必须对枚举字段做 case-insensitive 归一化，例如：

1. `button` -> `Button`  
2. `set_rect_anchored_position` -> `set_rect_anchored_position`（标准化小写）  
3. `validate`/`VALIDATE` -> `validate`

归一化失败才进入 schema 错误。

## 3. L3 执行层重构（非多态安全 DTO）

### 3.1 DTO 策略

L3 不再使用 `List<BaseType>` 多态反序列化。  
使用单请求 + 多具名数组强类型 DTO：

1. `List<HierarchyMutationDto>`  
2. `List<PropertyMutationDto>`  
3. `List<LayoutMutationDto>`  
4. `List<FileMutationDto>`

### 3.2 C# DTO 雏形

```csharp
[Serializable]
public sealed class ExecuteUnityTransactionRequestV2
{
    public string execution_mode;
    public string thread_id;
    public string idempotency_key;
    public string based_on_read_token;
    public string write_anchor_object_id;
    public string write_anchor_path;
    public List<string> execution_order;
    public List<HierarchyMutationDto> hierarchy_mutations;
    public List<PropertyMutationDto> property_mutations;
    public List<LayoutMutationDto> layout_mutations;
    public List<FileMutationDto> file_mutations;
}
```

执行顺序：

1. 按 `execution_order` 或默认固定顺序执行。  
2. 同一事务内统一 `UndoGuard` 包裹。  
3. 任一 mutation 失败即 Fail-Fast + 整体回滚。  

### 3.3 继续删除项

1. 删除 fuzzy 组件匹配与双源查找。  
2. 删除 `legacy_stringified_action_data/legacy_marshaled_action_data` 在新链路的消费。  
3. 删除隐式 fallback 修复逻辑。

## 4. 纵向契约统一（L1 -> L2 -> L3）

1. 全链路传输纯 JSON 对象，不再嵌套字符串化 JSON/Base64。  
2. 错误模型统一：`error_code`、`error_message`、`field_path`、`recoverable`。  
3. pending 语义统一：`pending_reason`、`last_stage`、`watchdog_deadline_ms`。

## 5. 分阶段实施路径（Milestones）

## Milestone 1：L1 契约冻结（聚合事务 + snapshot）

范围：

1. 落地 `execute_unity_transaction`。  
2. 落地 `get_scene_snapshot_for_write`。  
3. 落地 `get_write_contract_v2` full schema。  

验收门禁：

1. Sidecar 可编译。  
2. 新写工具 schema/示例测试全绿。  
3. 旧入口保留但冻结。  

## Milestone 2：L2 体验闭环（同步等待 + watchdog）

范围：

1. 线性 service flow 上线。  
2. 15 秒同步等待与 pending 诊断字段上线。  
3. watchdog 自动终止机制上线。  
4. 枚举大小写归一化上线。  

验收门禁：

1. 快操作默认返回终态比例达到目标（如 >= 95%）。  
2. 不再出现无原因长时 `pending`。  
3. 幂等/队列/OCC 回归测试全绿。  

## Milestone 3：L3 非多态 DTO 与严格执行

范围：

1. 多具名数组 DTO + 严格反序列化落地。  
2. Fail-Fast + Undo 原子回滚全链路打通。  
3. fuzzy/fallback 旧逻辑下线。  

验收门禁：

1. Unity EditMode 测试全绿。  
2. 原子回滚测试全绿。  
3. 错误码与字段路径稳定一致。  

## Milestone 4：默认切换与旧链路清除

范围：

1. 默认写路径切换到 `execute_unity_transaction`。  
2. 删除 `action_data*` 及旧写入口适配。  
3. 文档、示例、诊断脚本全部迁移。  

验收门禁：

1. E2E 回归全绿（写/回滚/并发/幂等）。  
2. pending 观测无黑洞样本。  
3. 代码库不再外露旧契约字段。
