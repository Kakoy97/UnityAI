# MCP 新增工具自动化开发方案（L1/L2/L3）

日期：2026-03-06  
状态：待实施（可直接进入开发）

## 1. 目标与边界

### 1.1 目标

把“新增一个 MCP 工具”的接入流程收敛为单一路径，做到：

1. L1 只改 SSOT 字典与编译器。
2. L2 不再手写 `definitions/*`、`*/validator.js`、`turnService` 一次性包装方法。
3. L3 不再手写 `SsotRequestDispatcher.cs` 的长链式 `if/switch` 分发分支。

### 1.2 非目标

1. 不改动通用基础设施协议：`/unity/query/pull`、`/unity/query/report`、OCC/Token 语义。
2. 不改变现有工具的业务行为与错误码口径（仅允许把实现入口迁移到自动化链路）。
3. 不处理“工具业务能力设计”本身（例如新 executor 的业务逻辑仍在 L3 实现）。

## 2. 当前基线（审计结论）

截至 2026-03-06，仓库内与“新增工具自动化”直接相关的存量耦合如下：

1. L2 手工定义文件：`sidecar/src/mcp/commands/definitions/*.js` 共 60 个。
2. L2 手工 validator 文件：`sidecar/src/mcp/commands/*/validator.js` 共 59 个。
3. L2 `turnService` 中 `dispatchSsotToolForMcp("...")` 一行式包装方法约 52 处。
4. L3 `SsotRequestDispatcher.cs` 中按工具名分发的 `if (string.Equals(...))` 分支约 48 处。
5. SSOT L2 产物 `mcp-tools.generated.json` 当前工具总数 60（含 2 个 deprecated）。

结论：L1 已有“字典 -> 产物”基础能力，但 L2/L3 仍存在大量手工桥接，新增工具尚未达成全自动接入。

## 3. 目标落地效果（验收口径）

完成后，新增一个新工具（非历史兼容工具）应只需要：

1. 在 `ssot/dictionary/tools.json` 增加定义。
2. 在 L3 新增/实现对应 executor。
3. 执行 `npm run ssot:build`。
4. 跑测试与编译验收。

且不再需要手改以下任何位置：

1. `sidecar/src/mcp/commands/definitions/*`
2. `sidecar/src/mcp/commands/*/validator.js`
3. `sidecar/src/application/turnService.js` 的工具包装方法
4. `Assets/Editor/Codex/Infrastructure/Ssot/SsotRequestDispatcher.cs` 的逐工具分发分支

## 6. 风险与控制

主要风险：

1. 自动化后工具可见性清单与安全 allowlist 不一致，导致误暴露。
2. L3 分发表迁移时，错误码口径漂移（尤其 unknown tool 与反序列化失败）。
3. 本地工具与 SSOT 工具混用导致入口判定错误。

控制措施：

1. 保留 deprecated/blocklist 强校验测试。
2. 固化 unknown tool 错误码回归测试：`E_SSOT_TOOL_UNSUPPORTED`。
3. 新增“工具分发模式白名单”测试（`ssot_query/local_static`）。

## 8. Step 6 详细设计补充（吸收评审）

本节用于补齐 L3 自动化实现细节，作为 Step 6 的强制设计约束。

### 8.1 Executor 创建与注入机制（P0）

结论：采用“工厂方法 + 字典延迟创建”，不采用运行时反射直接构造。

原因：

1. 反射路径在 Unity/AOT 场景下可控性与调试性较差。
2. 当前存在带依赖构造的 executor（例如 `ExecuteUnityTransactionSsotExecutor(Func<string,string,SsotDispatchResponse>)`），需要显式注入。
3. 工厂+缓存可以避免每次请求重复 new executor。

落地约束：

1. 新增 `ISsotExecutorFactory`（建议）作为统一创建入口。
2. 普通 executor 走默认无参构造；特殊 executor 在工厂中显式注册构造参数。
3. Dispatcher 内维护 `Dictionary<string, ISsotDispatchBinding>`，首次访问时通过工厂取实例并缓存。

### 8.2 类型转换与调用机制（P0）

结论：采用“编译期生成强类型 binding”，不采用每个工具手写 `as XxxRequestDto`。

绑定模型：

1. 每个工具生成一个 `ISsotDispatchBinding` 绑定项，内部固定：
   - DTO 反序列化入口（可复用 `SsotRequestRouter.TryDeserializeByToolName`）。
   - 强类型执行委托（`(executor, dto) => executor.Execute(dto)`）。
2. Dispatcher 只负责：
   - 读取 `tool_name`。
   - 查 binding。
   - 调用统一 `binding.TryExecute(...)`。

落地效果：

1. 新增工具不再改 `SsotRequestDispatcher.cs` 的类型转换分支。
2. 类型不匹配错误码口径统一由 binding 模板输出。

### 8.3 命名规范与特殊 Executor 规则（P1）

命名规范（强制）：

1. DTO：`<ToolNamePascalCase>RequestDto`
2. Executor：`<ToolNamePascalCase>SsotExecutor`
3. 绑定项：`<ToolNamePascalCase>DispatchBinding`（可生成）

特殊规则：

1. 若工具名与类名映射存在例外，必须在 SSOT 字典中声明显式别名字段（建议新增 `executor_class_name`）。
2. 需要依赖注入的 executor 必须在工厂中显式注册，禁止在 Dispatcher 中写特判构造逻辑。

### 8.4 迁移与回滚策略（P1）

采用“分发表双轨、执行单轨”：

1. 双轨对象仅限 Dispatcher 入口（legacy/generated 两套路由实现可切换）。
2. 同一请求只执行一条写链路，禁止写工具双执行对比。
3. 读工具可在灰度阶段做采样对比（可选）。

开关建议：

1. `SSOT_DISPATCHER_MODE=legacy|generated`
2. 默认 `legacy` 上线，测试环境与灰度环境先切 `generated`。
3. 当回归失败时可直接回滚到 `legacy`，不回滚业务代码。

### 8.5 性能与稳定性要求（P2）

1. Dispatcher 绑定表在域加载后一次性初始化，避免每请求反射扫描。
2. Executor 实例按需惰性创建并缓存。
3. 保留 `E_SSOT_TOOL_UNSUPPORTED` 与 `E_SSOT_DESERIALIZE_FAILED` 既有口径，不做语义漂移。

## 10. Step 8 提案：MCP 可见性白名单自动化（补充）

### 10.1 问题定义

当前 L2 运行时工具注册已经由 SSOT 产物驱动，但 MCP 可见性仍受以下静态合同约束：

1. `ROUTER_PROTOCOL_FREEZE_CONTRACT.mcp_tool_names`（手工列表）
2. `MCP_TOOL_VISIBILITY_FREEZE_CONTRACT.security_allowlist`（由上面列表展开）

这会导致“新增工具已完成自动注册，但未同步合同白名单时不可见”的问题，违背“新增工具只改字典+实现 executor”的目标。

### 10.2 目标

把 MCP 可见性策略收敛为“产物驱动 + 少量人工策略”，确保：

1. 新增 SSOT 工具默认可见（在 exposed 且非 deprecated/disabled 前提下）。
2. 不再维护手工 `mcp_tool_names/security_allowlist` 大列表。
3. 仅保留少量人工治理输入：`deprecated/disabled`（必要时）。

补充术语定义（强制）：

1. `exposed_tool`：以 `sidecar-command-manifest.generated.json` 的 `commands[*].name` 为准；即被注册到 MCP 命令体系的工具候选集合。
2. `deprecated_tool`：以 SSOT 字典 `lifecycle in ["deprecated","retired"]` 为准。
3. `local_static_tool`：以 `sidecar-command-manifest.generated.json` 中 `dispatch_mode === "local_static"` 为准，仅用于审计与测试，不参与 allowlist 判定。
4. `removed_tool`：以 SSOT 字典 `_definitions.removed_tool_names`（新增约束）为准；禁止继续以 `commandDefinitionManifest.js` 的运行时硬编码作为长期来源。
5. `active_tool`：`exposed_tool - deprecated_tool - removed_tool`；`disabled_tool` 在运行时再扣减。

### 10.3 方案决策（推荐）

采用“方案 A/C 合并”：**白名单自动生成并由产物驱动**，不采用“allowlist 为空即全放开”的运行时隐式策略。

原因：

1. 保留安全治理边界（显式 allowlist）而不是默认全开放。
2. 保持可审计（产物可比对、可快照测试）。
3. 与现有 SSOT 编译链一致，避免再引入新的手工源。

### 10.4 实施步骤

## Step 8.1（L1）：生成可见性快照产物

执行动作：

1. 在 SSOT 编译器新增 L2 emitter（建议名：`emitVisibilityPolicyManifest.js`）。
2. 产物以 Step 2 的 `sidecar-command-manifest.generated.json` 为主输入，再结合 SSOT 字典 lifecycle 与 `_definitions.removed_tool_names` 派生，不重复维护第二套来源。
3. 产物建议字段（固定结构）：
   - `version`
   - `generated_at`
   - `source`（dictionary/sidecar-manifest 路径与版本）
   - `removed_tool_names`
   - `exposed_tool_names`
   - `deprecated_tool_names`
   - `active_tool_names`
   - `local_static_tool_names`
4. 在 `ssot/compiler/index.js` 注册并输出到 `ssot/artifacts/l2/visibility-policy.generated.json`。
5. 为 `visibility-policy.generated.json` 定义编译期结构校验（JSON Schema 或等价断言），至少保证：
   - `removed_tool_names/deprecated_tool_names/active_tool_names/exposed_tool_names/local_static_tool_names` 均为字符串数组；
   - 集合去重后无空字符串；
   - `active_tool_names` 不包含 `removed_tool_names` 与 `deprecated_tool_names`。

产物结构示例（建议）：

```json
{
  "version": 1,
  "generated_at": "2026-03-06T10:00:00Z",
  "source": {
    "dictionary": "ssot/dictionary/tools.json",
    "sidecar_manifest": "ssot/artifacts/l2/sidecar-command-manifest.generated.json"
  },
  "removed_tool_names": ["instantiate_prefab"],
  "exposed_tool_names": ["create_object", "set_rect_anchored_position"],
  "deprecated_tool_names": ["get_prefab_info"],
  "active_tool_names": ["create_object", "set_rect_anchored_position"],
  "local_static_tool_names": ["get_tool_schema", "setup_cursor_mcp"]
}
```

完成标准：

1. `npm run ssot:build` 生成 `visibility-policy.generated.json`。
2. `exposed_tool_names` 与 `sidecar-command-manifest.generated.json` 的工具集合一一对应。
3. `deprecated_tool_names` 与字典 lifecycle(`deprecated/retired`) 一致。
4. `removed_tool_names` 与字典 `_definitions.removed_tool_names` 一致。
5. `active_tool_names = exposed - deprecated - removed` 可由测试重算通过。

## Step 8.2（L2）：contracts 去手工白名单

执行动作：

1. `sidecar/src/ports/contracts.js` 移除手工 `mcp_tool_names/security_allowlist` 大列表。
2. 改为在 Sidecar 启动时读取 `visibility-policy.generated.json` 形成 allowlist（运行时常量，不热更新）。
3. 明确失败策略：产物缺失/JSON 解析失败/字段格式非法时，**拒绝启动**（fail-fast），禁止静默 fallback 到“全开放”或“空列表”。
4. 保留 `disabled_tools` 作为人工治理入口；`deprecated` 来源改为产物（字典派生）而非手工列表。
5. 优先级规则：
   - `deprecated/removed`（产物）优先于其他策略，必须不可见；
   - `disabled_tools`（人工）在 active 集合上做运行时扣减。
6. 与 `startupArtifactsGuard` 集成（强制）：
   - 扩展 `assertSsotArtifactsAvailable`，新增 `visibility-policy.generated.json` 检查；
   - 新增 `DEFAULT_VISIBILITY_POLICY_PATH` 常量；
   - 在启动阶段统一调用 guard，失败直接终止启动流程（同现有 mcp-tools/ajv-schemas 口径）。

完成标准：

1. 新增工具后，不改 `contracts.js` 也可进入 MCP 可见集合（满足 exposed + not deprecated + not disabled）。
2. 旧口径错误码与 deprecated 拦截行为不变。
3. Sidecar 进程对“可见性产物不可用”具备可观测失败日志与明确错误码。

## Step 8.3（L2）：mcpServer 可见性判定改造

执行动作：

1. `sidecar/src/mcp/mcpServer.js` 使用统一公式：
   - `visible = registry_exposed ∩ policy.active_tool_names - disabled_tools`
   - 同时对 `deprecated/removed` 做硬拦截（错误口径保持不变）。
2. `isToolVisibleByPolicy` 保持单一出口，不在其他位置重复策略判断。
3. 改造 `isToolAllowedByVisibilityContract` 的实现来源：
   - 移除对静态 `MCP_SECURITY_ALLOWLIST_TOOL_NAME_SET` 的依赖；
   - 改为读取启动期加载后的 `policy.active_tool_names`；
   - 继续保留 `deprecated/removed/disabled` 判定并执行固定优先级。

完成标准：
