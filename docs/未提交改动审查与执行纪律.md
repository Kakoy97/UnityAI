# 未提交改动审查与执行纪律

## 1. 目的与背景
当前工作区存在大量未提交变更。为避免在 L1/L2 旧链路清除过程中误触基础底座（门禁、通用运行时、SSOT 主链路），本文件定义统一审查流程与执行纪律。

本文件适用于以下场景：
1. 审查 `git status` 中未提交改动是否偏离“旧链路清除”目标。
2. 分批推进清理任务并控制回归风险。
3. 在“需要改动新链路功能”时提供升级通知与审批触发点。

## 2. 审查目标
1. 识别并清理 L1/L2 旧链路残留，避免保留双轨入口。
2. 保证 SSOT 新链路不被破坏：`turnServiceMethod + validate + dispatchSsotToolForMcp`。
3. 保证基础底座能力不回退：token/revision/schema 门禁、query 运行时、核心错误反馈口径。
4. 输出可追溯证据：文件级变更、测试门禁结果、残留风险清单。

## 3. 范围与非目标
### 3.1 范围（本轮允许）
1. `sidecar/src/api`
2. `sidecar/src/application`
3. `sidecar/src/mcp`
4. `sidecar/src/ports`
5. 对应测试与 `docs` 状态回填

### 3.2 非目标（本轮禁止）
1. 修改 Unity L3 执行器实现逻辑。
2. 修改 SSOT 字典编译产物协议字段定义。
3. 在无通知情况下新增“旧 -> 新”或“新 -> 旧”的兼容桥接。

## 4. 执行纪律（硬约束）
1. 只在“审查白名单路径”内改动；出现白名单外改动必须立即停止并报告。
2. 每批改动必须先有“引用证据”再删除代码，禁止盲删。
3. 每批完成后必须跑门禁测试，未通过不得进入下一批。
4. 若发现“旧链路清除需要改新链路功能”，必须先通知并得到确认，再实施。
5. 禁止为通过测试而回补旧入口或回退到 legacy fallback。

## 5. 分批审查步骤
### Step 0：基线冻结
1. 记录当前分支与工作区状态：
   - `git rev-parse --abbrev-ref HEAD`
   - `git status --short`
2. 输出本轮审查目标与白名单。

### Step 1：构建调用图与入口图
1. 枚举 Router 活跃入口与废弃入口。
2. 枚举 command definitions 的 `turnServiceMethod`。
3. 枚举 `turnService` 方法并统计全仓引用（src + tests）。

### Step 2：分级判定
对每个候选方法进行三级判定：
1. `KEEP`：新链路或底座必需，禁止删。
2. `MIGRATE`：仍有测试/工具依赖，先迁移调用口径再删。
3. `REMOVE`：无生产调用，仅兼容残留，可删除。

### Step 3：实施与同步
1. 先删 `REMOVE` 项。
2. 对 `MIGRATE` 项先迁移调用，再删旧方法。
3. 同步更新 `docs/SSOT 字典架构方案与旧代码清退计划.md` 的 L1/L2 状态口径。

### Step 4：门禁回归
每批至少执行以下门禁：
1. `node --test sidecar/tests/application/protocol-write-consistency.test.js`
2. `node --test sidecar/tests/application/ssot-batch-route.test.js`
3. `node --test sidecar/tests/application/ssot-validator-registry.test.js`
4. `node --test sidecar/tests/application/ssot-token-registry.test.js`
5. `node --test sidecar/tests/application/ssot-revision-state.test.js`

按改动内容追加：
1. 若涉及 job/lease：`job-lease-janitor.test.js`
2. 若涉及观测口径：`error-feedback-metrics-wiring.test.js`、`r20-governance-metrics-wiring.test.js`
3. 若涉及写入门禁：`occ-write-guard.test.js`、`mcp-write-readiness-and-error-taxonomy.test.js`

### Step 5：验收与归档
1. 输出“已清理项 / 保留项 / 待确认项”三栏清单。
2. 附门禁结果（通过/失败 + 错误摘要）。
3. 在计划文档中回填日期与状态标签（`已完成/进行中/待确认`）。

## 6. 何时必须先通知再改
出现以下情况必须先通知：
1. 需要调整 SSOT 请求/响应结构字段。
2. 需要修改 token/revision/schema 门禁判定逻辑。
3. 需要变更核心底座行为（query coordinator、turn runtime、全局错误分类）。
4. 需要新增跨层桥接（L2 与 L3 之间新适配层）。

通知模板（简版）：
1. 触发原因
2. 受影响模块
3. 不改会怎样
4. 改动方案与回滚点

## 7. 审查输出模板
每批输出固定结构：
1. 范围：本批涉及文件列表。
2. 结论：`KEEP/MIGRATE/REMOVE` 统计。
3. 改动：已删除旧链路项与迁移项。
4. 风险：潜在影响与是否触发“先通知”条件。
5. 验证：门禁命令与结果。
6. 文档回填：已更新的计划文档与行号。

## 8. 完成定义（DoD）
1. Router/Definition/TurnService 无旧入口回流。
2. SSOT 主链路可用，核心门禁测试通过。
3. 文档口径与代码状态一致。
4. 未出现白名单外改动。
