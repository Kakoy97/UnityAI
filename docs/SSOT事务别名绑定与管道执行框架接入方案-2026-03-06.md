# SSOT 事务别名绑定与管道执行框架接入方案（L1/L2/L3）
日期：2026-03-06  
状态：设计完成，待按阶段实施

## 1. 目标与约束
### 1.1 目标
将 `execute_unity_transaction` 从“字符串步骤列表”升级为“框架级事务执行模型”，满足：

1. 大模型可一次性规划全流程，执行器可在同一事务中完成多步写操作。
2. 支持步骤输出别名与后续步骤引用，解决 `create_object` 结果（`target_object_id/target_path`）无法继续消费的问题。
3. 在不引入胶水代码的前提下实现高解耦、可扩展、单一职责。
4. 保持 SSOT 主路径：L1 字典定义 -> L2 校验与路由 -> L3 强类型执行。

### 1.2 非目标
1. 不保留“最小补丁”式字符串正则替换方案。
2. 不新增“工具特判 if/switch”到事务执行器内部。
3. 不改变非事务工具的业务语义与错误码口径（除事务接入新增错误码）。

### 1.3 改动边界（防止误伤底座）
1. 本方案只允许改动 `ssot/`、`sidecar/src/application/ssotRuntime/`、`sidecar/src/application/turnService.js`、`sidecar/src/ports/contracts.js`、`Assets/Editor/Codex/Infrastructure/Ssot/`、`Assets/Editor/Codex/Generated/Ssot/`。
2. 本方案不改动基础设施底座：`mcpGateway` 网络入口、门禁系统、鉴权、流式通道、job/metrics 基础框架、通用日志与监控管线。
3. 若实施中发现必须触碰底座，必须先新增“变更通知条目”，经确认后再执行，不得直接落盘。

## 2. 现状问题（必须一次性解决）
当前事务能力存在结构性缺口：

1. `execute_unity_transaction` 仅支持 `steps_json` 字符串，步骤结构为 `tool_name + payload_json`，缺少依赖/别名语义。
2. `CreateObject` 步骤虽然返回 `target_object_id/target_path`，但无法被后续步骤结构化引用。
3. 当前执行器允许任意步骤工具名，缺少事务安全策略层。
4. 步骤 payload 走字符串，不具备可组合扩展能力（难以演进到依赖图、条件执行、结果绑定）。

关键现有实现：

1. L3 事务执行器：`Assets/Editor/Codex/Infrastructure/Ssot/Executors/ExecuteUnityTransactionSsotExecutor.cs`
2. L1 字典（现有 `steps_json`）：`ssot/dictionary/tools.json`
3. L3 DTO 生成器（目前对象/数组对象映射能力不足）：`ssot/compiler/emitters/l3/emitDtosCs.js`

## 3. 目标架构（框架级，不做补丁）
## 3.1 事务合同 V2（L1 SSOT）
将 `execute_unity_transaction` 输入从：

```json
{
  "transaction_id": "txn_xxx",
  "steps_json": "[{\"tool_name\":\"create_object\",\"payload_json\":\"...\"}]"
}
```

升级为结构化合同：

```json
{
  "transaction_id": "txn_xxx",
  "steps": [
    {
      "step_id": "step_create_button",
      "tool_name": "create_object",
      "payload": { "...": "..." },
      "save_as": "button",
      "depends_on": []
    },
    {
      "step_id": "step_set_button_pos",
      "tool_name": "set_rect_anchored_position",
      "payload": {
        "target_object_id": { "$ref": "button.target_object_id" },
        "target_path": { "$ref": "button.target_path" },
        "x": -150,
        "y": -50
      },
      "depends_on": ["step_create_button"]
    }
  ]
}
```

说明：

1. `save_as` 负责输出命名。
2. `$ref` 只允许“整值引用”，不允许字符串拼接模板。
3. `depends_on` 支持未来 DAG 扩展（首版可先仅允许引用已执行步骤，保持串行执行）。
4. `steps[*].payload` 定义为“业务字段载荷”，不允许出现 `execution_mode/idempotency_key/based_on_read_token/write_anchor_*` 等 `write_envelope` 字段。
5. `save_as` 保存的是该步骤 `SsotDispatchResultData` 的快照；`$ref` 语法为 `{ "$ref": "<alias>.<field>" }`，例如 `button.target_object_id`。
6. 首版 `$ref` 支持字段白名单：`target_object_id`、`target_path`、`target_object_name`、`scene_revision`、`value_kind`、`value_string`、`value_number`、`value_boolean`；超出白名单返回 `E_TRANSACTION_REF_PATH_INVALID`。
7. `depends_on` 首版允许多个依赖，但每个依赖必须指向“前置步骤 step_id”；执行顺序仍严格按数组顺序串行，`depends_on` 仅用于合法性校验与可读性。
8. `save_as` 是可选字段；未提供 `save_as` 的步骤不可被后续 `$ref` 引用。
9. `save_as` 在同一事务内必须唯一；重复定义由 `TransactionPlanValidator` 拦截并返回 `E_TRANSACTION_PLAN_INVALID`。

## 3.2 分层职责（单一职责）
### L1（定义与生成）
1. 只定义事务协议与校验结构，不承担执行逻辑。
2. 负责生成 L2 AJV 与 L3 DTO/Binding 所需元数据。

### L2（路由与门禁）
1. 负责 schema 门禁与事务策略门禁（工具可达性、生命周期、禁用策略）。
2. 不做步骤业务执行，不解析 Unity 对象语义。

### L3（执行引擎）
1. 只处理事务执行域逻辑：解析计划、依赖检查、引用解析、逐步执行、回滚控制。
2. 不关心 MCP 路由、不关心 HTTP 协议。

## 3.3 L3 模块拆分（拒绝胶水）
新增事务域模块（建议目录：`Assets/Editor/Codex/Infrastructure/Ssot/Transaction/`）：

1. `TransactionPlanModel.cs`
职责：事务步骤、引用、依赖关系的数据模型。

2. `TransactionPlanValidator.cs`
职责：步骤唯一性、依赖合法性、循环依赖检测、`save_as` 唯一性校验。

3. `TransactionSafetyPolicy.cs`
职责：校验步骤工具是否允许在事务中执行（禁止 deprecated/removed/禁用/非事务安全工具）。

4. `TransactionAliasStore.cs`
职责：管理 `save_as -> step result` 的变量存储。

5. `TransactionReferenceResolver.cs`
职责：解析 `$ref` 到实际值并输出步骤 payload（无正则拼接）。

6. `TransactionExecutionEngine.cs`
职责：串行执行步骤、收集中间结果、失败回滚、构建事务结果。

7. `ExecuteUnityTransactionSsotExecutor.cs`（保留为 orchestration facade）
职责：只做调用编排，不持有具体规则。

## 3.4 架构守则（强约束）
1. 禁止在 `ExecuteUnityTransactionSsotExecutor` 中写工具名特判分支；工具级策略只能在 `TransactionSafetyPolicy`。
2. 禁止在 L2 `turnService` 拼接或改写步骤 payload；L2 只校验与路由。
3. 禁止在 L1 字典中嵌入运行时策略代码；L1 仅提供 schema 与生命周期元数据。
4. 引用解析必须集中在 `TransactionReferenceResolver`，不得散落在执行器和工厂类。
5. 事务失败处理必须集中在 `TransactionExecutionEngine`，不得由各工具执行器重复实现回滚编排。

## 3.5 事务上下文与 Token 语义（新增）
1. `execute_unity_transaction` 作为 write 工具，沿用现有 `write_envelope`；L2 仅在“事务入口”校验一次 `based_on_read_token`。
2. 入口 token 校验失败继续沿用现有全局错误码：`E_TOKEN_UNKNOWN`、`E_TOKEN_EXPIRED`、`E_SCENE_REVISION_DRIFT`，不新增分叉错误码。
3. 事务内部步骤不重复做 token 校验，避免“步骤 1 改变 scene_revision 导致步骤 2 必然失效”的伪冲突。
4. L3 在事务启动时构建 `TransactionExecutionContext`（`execution_mode`、`idempotency_key`、`based_on_read_token`、`write_anchor_object_id`、`write_anchor_path`），每个步骤执行前自动注入该上下文。
5. 步骤 payload 经过 `TransactionReferenceResolver` 解析后必须是“纯值 JSON”（不再含 `$ref` 对象）再进入具体工具 DTO 反序列化。
6. `TransactionExecutionContext` 字段全部来自事务入口 DTO 顶层字段，不从步骤结果推断、不做隐式兜底。
7. `write_anchor_object_id/write_anchor_path` 在事务入口必须有效；缺失或空值直接失败（L2 schema 或 L3 `E_TRANSACTION_CONTEXT_INVALID`），不允许“从第一步 `save_as` 回填”。

## 3.6 回滚边界与性能边界（新增）
1. 首版回滚模型：`Undo.IncrementCurrentGroup` + 事务组内执行 + 失败时 `Undo.RevertAllDownToGroup` + 成功时 `Undo.CollapseUndoOperations`。
2. 非 Undo 安全工具必须禁止进入事务步骤（例如 `save_scene`、`save_prefab`），由 `TransactionSafetyPolicy` 拦截。
3. 首版事务步骤只允许 `write` 工具，不允许 `read` 工具；`read` 工具由 `TransactionSafetyPolicy` 直接拦截并返回 `E_TRANSACTION_STEP_TOOL_FORBIDDEN`。原因：首版事务语义定义为“可回滚写集”，不引入读写混合失败语义。
4. 首版执行保持串行，不做并行步骤调度；原因是 Unity Editor API 主线程约束 + 原子回滚一致性优先。
5. 性能收益来自“减少 MCP 往返和中间读 token 请求”，不是来自 Unity 内部并行执行。
6. 未来优化方向仅作为后续版本：基于依赖图的并行可行性评估、批量同构写入折叠，不纳入本期 DoD。

## 3.7 `$ref` 解析边界（新增）
1. 解析顺序：先做计划静态校验（别名定义、字段白名单、依赖合法性），再按步骤执行时解析具体值。
2. `alias` 不存在：返回 `E_TRANSACTION_ALIAS_MISSING`。
3. `field` 非白名单：返回 `E_TRANSACTION_REF_PATH_INVALID`。
4. `field` 值为 `null`：允许替换为 `null`；后续是否可通过由目标工具 schema 决定。
5. `field` 值为对象/数组：首版不支持将对象/数组作为 `$ref` 结果值，返回 `E_TRANSACTION_REF_PATH_INVALID`。
6. `$ref` 标记允许出现在 payload 任意层级（对象/数组）中，但必须是“整值对象”且仅包含一个 `$ref` 字段。
7. 不支持字符串内插（例如 `"id=$ref:xxx"`）或与其他键混合的“半结构化 `$ref`”对象。
8. 任一步解析失败立即中断事务并回滚已执行步骤。

## 4. 分阶段落地步骤（可执行）
## Step A（L1）：升级事务协议
执行动作：

1. 改造 `ssot/dictionary/tools.json` 中 `execute_unity_transaction` 输入，从 `steps_json` 切换为结构化 `steps`。
2. 增加事务步骤 schema 复用定义（推荐放 `_definitions`，供后续工具重用）。
3. 为 `$ref` 增加 schema 约束（对象形态、字段约束）。
4. 在字典中新增事务策略元数据（建议字段：`transaction.enabled`、`transaction.undo_safe`），并作为 L2/L3 统一策略来源，禁止手写硬编码名单。
5. 首版在字典层显式将 `save_scene`、`save_prefab` 标记为 `transaction.enabled=false`。
6. 明确元数据挂载位置：每个工具对象顶层（与 `name/kind/lifecycle` 同级），结构如下：
   ```json
   {
     "name": "create_object",
     "kind": "write",
     "transaction": {
       "enabled": true,
       "undo_safe": true
     }
   }
   ```
7. 首版默认策略采用“默认拒绝”：
   - `transaction.enabled` 缺省视为 `false`
   - `transaction.undo_safe` 缺省视为 `false`
   - 只有显式 `enabled=true && undo_safe=true` 的工具可进入事务
8. 策略优先级（从高到低）：
   - `removed/deprecated`（生命周期拦截）
   - `transaction.enabled/undo_safe`（事务准入）
   - 运行时 disabled 策略（人工禁用）
   - 其他可见性与路由条件
9. 迁移策略：Step A 一次性为所有 write 工具补齐 `transaction` 元数据，禁止“边改边漏”。
10. 首版准入清单规则：`kind=write` 且 `lifecycle=stable/experimental` 的工具，需显式标注 `transaction.enabled=true/false` 与 `transaction.undo_safe=true/false`。
11. 构建守护：新增字典完整性校验，若发现 write 工具缺失 `transaction` 元数据则 `ssot:build` 失败（fail-fast），不允许静默拒绝。

涉及文件：

1. `ssot/dictionary/tools.json`

验收标准：

1. `npm run ssot:build` 成功。
2. `mcp-tools.generated.json` 与 `ajv-schemas.generated.json` 出现新 `steps` 结构。

6. 明确执行流程：`校验计划 -> 校验策略 -> 解析 $ref -> 注入上下文 -> 调度步骤 -> 记录 save_as -> 下一步`。
7. `$ref` 解析发生在步骤调度前；调度 payload 必须为无 `$ref` 的最终 JSON。
8. 增加失败分类与透传：
   - 计划/引用阶段失败：返回事务级错误码（如 `E_TRANSACTION_PLAN_INVALID/E_TRANSACTION_ALIAS_MISSING/E_TRANSACTION_REF_PATH_INVALID`）
   - 步骤执行失败：顶层返回 `E_TRANSACTION_STEP_FAILED`，并透传原始 `failed_error_code/failed_error_message`
9. 建议实现伪代码：
   ```text
   buildContext(request)
   validatePlan(plan)
   beginUndoGroup()
   for step in steps:
     resolvedPayload = resolveRefs(step.payload, aliasStore)
     finalPayload = injectWriteEnvelope(resolvedPayload, context)
     result = dispatch(step.tool_name, finalPayload)
     if failed(result): rollback + return stepFailedEnvelope
     if step.save_as exists: aliasStore[step.save_as] = result.data
   collapseUndoGroup()
   return success
   ```

## 6. 错误码与可观测性（新增）
建议新增并固定：

1. `E_TRANSACTION_PLAN_INVALID`（计划结构非法）
2. `E_TRANSACTION_DEPENDENCY_CYCLE`（依赖图循环）
3. `E_TRANSACTION_DEPENDENCY_ORDER_INVALID`（依赖指向后置步骤或不存在步骤）
4. `E_TRANSACTION_ALIAS_MISSING`（别名不存在）
5. `E_TRANSACTION_REF_PATH_INVALID`（引用字段非法）
6. `E_TRANSACTION_STEP_TOOL_FORBIDDEN`（步骤工具被策略禁止）
7. `E_TRANSACTION_STEP_FAILED`（步骤执行失败并回滚）
8. `E_TRANSACTION_CONTEXT_INVALID`（事务上下文缺失，无法注入步骤 envelope）

返回体建议包含：

1. `failed_step_index`
2. `failed_step_id`
3. `failed_tool_name`
4. `rollback_applied`
5. `failed_error_code`
6. `failed_error_message`
7. `resolved_ref_count`
8. `executed_step_count`

建议失败返回体（示例）：
```json
{
  "error_code": "E_TRANSACTION_STEP_FAILED",
  "error_message": "Step 'step_set_button_pos' failed",
  "failed_step_index": 1,
  "failed_step_id": "step_set_button_pos",
  "failed_tool_name": "set_rect_anchored_position",
  "failed_error_code": "E_SSOT_SCHEMA_INVALID",
  "failed_error_message": "target_object_id is required",
  "rollback_applied": true,
  "resolved_ref_count": 2,
  "executed_step_count": 1
}
```

## 7. 迁移与清理策略（非补丁式）

3. 删除旧字符串步骤解析逻辑与相关测试夹层，保持唯一主链。

## 8. Definition of Done
同时满足以下条件才算完成：

1. `execute_unity_transaction` 已使用结构化步骤 + alias/ref，不再依赖 `steps_json` 字符串。
2. 事务内部可完成“创建对象 -> 后续步骤引用对象锚点”闭环。
3. 事务步骤工具受统一策略门禁，不可绕过 deprecated/removed/disabled。
4. L1/L2/L3 各层职责清晰，无工具特判胶水代码。
5. Node + Unity 测试与编译全绿，且旧路径已具备可删除计划。
6. 底座基础设施目录无改动，或存在经确认的变更通知记录。
7. 事务使用者无需在每个步骤重复提供 token 与 write_envelope 字段，步骤可直接消费 `save_as/$ref`。

## 10. 二次评审结论（对照 Cursor 7 项）
1. Token 管理机制不明确：采纳。已在 3.5 明确“入口校验一次 + 步骤内不重复校验 + 上下文自动注入”。
2. `$ref` 解析时机与 DTO 冲突：采纳。已在 3.5 与 Step D 明确“调度前解析为纯值 JSON”。
3. `write_envelope` 步骤字段处理不明确：采纳。已在 3.1 与 Step D 明确“步骤只写业务字段，由事务上下文自动注入”。
4. `save_as` 输出结构不明确：采纳。已在 3.1 明确“保存 `SsotDispatchResultData` 快照 + `$ref` 路径规则 + 白名单字段”。
5. 回滚机制细节不足：采纳。已在 3.6 明确 Undo 组边界与非 Undo 安全工具禁入。
6. 依赖与执行顺序不明确：采纳。已在 3.1、Step E 明确“多依赖允许、仅可依赖前置步骤、执行仍串行”。
7. 性能优化方向缺失：部分采纳。已补充 3.6 的性能边界与后续优化方向；本期不引入并行执行。

补充说明：
1. 未采纳“新增 `E_TRANSACTION_TOKEN_INVALID`”。原因：当前全局 token 错误口径已稳定，继续复用 `E_TOKEN_UNKNOWN/E_TOKEN_EXPIRED/E_SCENE_REVISION_DRIFT`，避免错误码分叉。

## 11. 三次评审结论（对照 Cursor 新 5 项）
1. 事务策略元数据结构不明确：采纳并修正。已在 Step A 明确字段挂载位置、结构样例、默认值与优先级。
2. `TransactionExecutionContext` 实现细节不明确：采纳并修正。已在 3.5 明确来源与必填约束，并在 Step D 增加执行伪代码。
3. `$ref` 边界情况不明确：采纳并修正。已新增 3.7，覆盖 alias 缺失、字段非法、`null` 值、对象/数组值限制。
4. 步骤失败错误信息详细度不足：采纳并修正。已在 Step D 与第 6 节补充失败分类、原始错误透传与示例返回体。
5. `save_as` 可选性不明确：采纳并修正。已在 3.1 明确“可选、不可引用无别名步骤、别名必须唯一”。

补充说明：
1. 未采纳“事务策略默认全放开（`enabled=true`）”。为避免误将非 Undo 安全工具纳入事务，采用默认拒绝、显式准入。
